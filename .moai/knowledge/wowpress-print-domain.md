## 1. 개요

### 1.1 문서 목적

이 문서는 인쇄 자동견적 위젯 빌더를 개발하기 위한 **완전한 도메인 지식 참조 문서**이다. 와우프레스(WowPress)의 실제 API 데이터와 제품 스펙 문서를 분석하여, 인쇄 산업의 주문 옵션 체계, 제약 조건 시스템, 가격 계산 모델을 체계적으로 정리하였다.

### 1.2 데이터 출처

| 출처 | 설명 | 비고 |
|------|------|------|
| WowPress Open API v0.0.1 | REST API (api.wowpress.co.kr) | JWT 인증, 기준정보 + 주문/가격 |
| products_spec_v1.0.pdf | 제품상세 SPEC (24페이지) | 기준정보 JSON 구조 정의 |
| price_order_spec_v1.01.pdf | 가격/주문 SPEC (25페이지) | 가격 조회, 주문 프로세스 |
| 카탈로그 JSON | 326개 상품 실제 데이터 | ref/wowpress/catalog/ |
| 공개 사이트 UI 분석 | wowpress.co.kr 주문 페이지 | jQuery + AJAX 기반 |

### 1.3 핵심 발견

모든 와우프레스 상품은 **동일한 JSON 구조**를 공유한다. 상품 간 차이는 카테고리, 옵션 종류, 제약 조건의 **값(value)** 뿐이다. 따라서 하나의 상품을 완벽히 이해하면 전체 326개 상품을 처리할 수 있는 범용 엔진을 구축할 수 있다.

## 2. 상품 구조 (Product Structure)

### 2.1 JSON 구조 개요

`GET /api/v1/std/prod_info/{prodno}` API가 반환하는 제품 상세 JSON의 최상위 구조이다.

```
prod_info
  +-- prodno, prodname, seltype, pjoin, unit, ctptime, filetype
  +-- dlvyprepay, dlvygrpno, dlvygrpname
  +-- coverinfo[]          ... 커버/페이지 구성
  +-- ordqty[]              ... 주문 가능 수량
  +-- sizeinfo[]            ... 규격 정보
  +-- paperinfo[]           ... 지류 정보
  +-- colorinfo[]           ... 도수 정보
  +-- optioninfo[]          ... 추가 옵션 (MD 상품)
  +-- prsjobinfo[]          ... 인쇄방식 프리셋
  +-- awkjobinfo[]          ... 후가공 정보 (가장 복잡)
  +-- prodaddinfo[]         ... 부자재 정보
  +-- deliverinfo{}         ... 배송 정보
```

### 2.2 기본 필드 설명

| 필드 | 타입 | 설명 | 특수지명함 값 |
|------|------|------|-------------|
| prodno | Int | 상품 고유번호 | 40070 |
| prodname | String | 상품명 | "특수지명함" |
| seltype | String | 제품타입 ("M"이 대부분) | "M" |
| pjoin | Int | 표지/내지 구분 | 0 (통합) |
| unit | String | 수량 단위 | "매" |
| ctptime | String | 조판시작시간 | null |
| dlvyprepay | Boolean | 선불택배 가능 여부 | true |
| dlvygrpno | Int | 묶음배송 그룹 ID | 1 |
| dlvygrpname | String | 묶음배송 그룹명 | "명함,스티커 묶음" |
| filetype | String | 허용 파일형식 (쉼표 구분) | "AI,EPS,JPG,PNG,CDR,PSD,SIT,ZIP,ALZ,PDF" |

### 2.3 pjoin 유형

| pjoin | 구분 | 설명 | 대표 상품 |
|-------|------|------|---------|
| 0 | 통합 | 표지/내지 구분 없는 단일 인쇄물 | 명함, 스티커, 전단, 포스터 |
| 1 | 분리 | 표지(covercd=1) + 내지(covercd=2) + 간지(covercd=3) | 무선책자, 중철책자 |

### 2.4 coverinfo (커버/페이지 구성)

제품의 페이지 레이어 구조를 정의하는 최상위 설정이다.

| covercd | 구분명 | 설명 | pagecd | 페이지 구분 |
|---------|------|------|--------|-----------|
| 0 | 통합 | 단일 인쇄물 | 0=양면, 1=전면, 2=후면 |
| 1 | 표지 | 책자 겉표지 | 0=양면, 1=전면, 2=후면 |
| 2 | 내지 | 책자 속지 | 0=양면, 1=전면, 2=후면 |
| 3 | 간지 | 책자 중간 구분지 | 0=양면 |

특수지명함의 실제 coverinfo:
```json
{
  "covercd": 0,
  "covername": "통합",
  "pagelist": [{"pagecd": 0, "pagename": "양면"}],
  "pagecnt": {"min": null, "max": null, "interval": null}
}
```

pagecnt의 min/max가 null이면 페이지수 선택이 불필요한 상품이다. 책자류(pjoin=1)에서는 min=8, max=500, interval=4 등의 값이 설정된다.

## 3. 기준정보 우선순위 체인 (Option Priority Chain)

### 3.1 체인 구조

모든 상품의 옵션 선택은 아래 고정된 우선순위를 따른다.

```
jobpresetno (인쇄방식 프리셋) [우선순위 1]
  -> sizeno (규격정보) [우선순위 2]
    -> paperno (지류정보) [우선순위 3]
      -> optno (옵션정보) [우선순위 4]
        -> colorno (도수정보) [우선순위 5]
          -> colornoadd (추가도수정보) [우선순위 6]
```

### 3.2 핵심 규칙

- **역방향 선택 금지**: 규격을 선택하지 않은 상태에서 지류를 먼저 선택할 수 없다
- **상위 변경 시 하위 초기화**: 규격을 변경하면 지류/도수/수량 선택이 리셋된다
- **단일 프리셋 상품**: 인쇄방식이 1개뿐인 상품(대부분)은 자동 선택되므로 UI에서 생략 가능

### 3.3 수량 결정의 두 가지 유형

| 유형 | 조건 | 설명 | 대표 상품 |
|------|------|------|---------|
| 기준정보 공통형 | ordqty의 모든 기준정보 키가 null | 어떤 조합이든 동일한 수량 목록 | 가성비스티커 |
| 기준정보 조합형 | ordqty에 sizeno, paperno 등이 지정 | 규격+지류 조합별로 다른 수량 | 특수지명함, 롤스티커 |

특수지명함은 **조합형**이다. sizeno + paperno 조합에 따라 수량 목록이 달라진다.

## 5. 제약 조건 시스템 (Constraint System)

### 5.1 제약 조건 개요

모든 기준정보 간의 의존 관계는 두 가지 접두사로 표현된다.

- **req_* (required)**: 선택 시 다른 기준정보가 **활성화/필수**가 되는 조건
- **rst_* (restricted)**: 선택 시 다른 기준정보가 **비활성화/금지**되는 조건

값이 null이면 해당 제약이 없다는 의미이다.

### 5.2 제약 조건 종류별 상세

#### req_* (요구사항) 계열

| 제약 | 발생 위치 | 의미 | 특수지명함 실례 |
|------|---------|------|-------------|
| req_width | sizeinfo | 비규격 가로 입력범위 (min/max/interval) | sizeno=7071: min=35, max=297 |
| req_height | sizeinfo | 비규격 세로 입력범위 | sizeno=7071: min=35, max=450 |
| req_prsjob | colorinfo | 이 도수를 선택하면 해당 인쇄방식만 허용 | colorno=255,256 -> jobpresetno=3110 |
| req_awkjob | sizeinfo, paperinfo, colorinfo | 이 옵션 선택 시 특정 후가공이 필수 활성화 | (특수지명함에는 해당 없음) |
| req_joboption | awkjobinfo | 후가공 선택 시 추가 옵션 입력 필요 | 부분라운딩 -> 좌상/우상/좌하/우하 선택 |
| req_jobsize | awkjobinfo | 후가공 선택 시 작업 규격 입력 필요 | (특수지명함에는 해당 없음) |
| req_jobqty | awkjobinfo | 후가공 선택 시 작업 수량 입력 필요 | (특수지명함에는 해당 없음) |
| req_color | prsjobinfo | 이 인쇄방식에서 가능한 도수 목록 | (특수지명함에는 해당 없음, 단일 프리셋) |

#### rst_* (제약사항) 계열

| 제약 | 발생 위치 | 의미 | 특수지명함 실례 |
|------|---------|------|-------------|
| rst_ordqty | sizeinfo, paperinfo | 이 옵션 선택 시 수량 범위 제한 | (특수지명함에는 해당 없음) |
| rst_awkjob | sizeinfo, paperinfo, colorinfo | 이 옵션 선택 시 특정 후가공 금지 | sizeno=5457 -> 오시/미싱 크기 제한, paperno=20426 -> 박 72개 금지 |
| rst_prsjob | paperinfo | 이 지류 선택 시 특정 인쇄방식 금지 | (특수지명함에는 해당 없음) |
| rst_jobqty | awkjobinfo | 이 후가공은 특정 수량 범위에서만 가능 | 오시: 100~10,000매, 박: 100~2,000매 |
| rst_cutcnt | awkjobinfo | 이 후가공은 특정 컷수 이상에서만 가능 | 미싱2줄: min=1, 미싱3줄: min=2 |
| rst_size | awkjobinfo | 이 후가공은 특정 규격 크기에서만 가능 | 오시(가로): w=70~300, h=100~700 |
| rst_paper | awkjobinfo | 이 후가공은 특정 지류에서 사용 불가 | 박 전체: 스코틀랜드 220g 금지 |
| rst_color | awkjobinfo | 이 후가공은 특정 도수에서 사용 불가 | (특수지명함에는 해당 없음) |

### 5.3 특수지명함의 전체 제약 조건 맵

```
규격 88x53mm (sizeno=5457)
  -> [rst_awkjob] 가로 오시: 가로 70~300 범위 안 -> 허용
  -> [rst_awkjob] 세로 오시: 세로 53 < 70 최소 -> 금지
  -> [rst_awkjob] 미싱 전체: 가로 88 < 100 최소 -> 금지

스코틀랜드 220g (paperno=20426)
  -> [rst_awkjob] 박(앞면) 36개 + 박(뒷면) 36개 = 72개 금지

도수 단면4도 / 양면8도
  -> [req_prsjob] 옵셋인쇄만 허용

오시 후가공 선택 시
  -> [rst_jobqty] 100~10,000매 범위에서만 가능
  -> [rst_size] 가로/세로 크기 범위 체크

박 후가공 선택 시
  -> [rst_jobqty] 100~2,000매 범위에서만 가능
  -> [rst_paper] 스코틀랜드 220g 금지

타공 후가공 선택 시
  -> [rst_jobqty] 100~10,000매 범위에서만 가능
  -> [rst_cutcnt] 3개 타공은 최소 2컷 필요

형압 후가공 선택 시
  -> [rst_jobqty] 100~10,000매 범위에서만 가능

라운딩 후가공 선택 시
  -> [rst_jobqty] 최소 100매
  -> [req_joboption] 부분라운딩은 모서리 1~3개 선택 필요
```

### 5.4 제약 조건 평가 알고리즘 (구현 시 참고)

```
1. 사용자가 기준정보를 선택한다
2. 선택된 기준정보의 req_* 조건을 수집한다
   -> 필수 활성화 대상 목록 생성
3. 선택된 기준정보의 rst_* 조건을 수집한다
   -> 비활성화 대상 목록 생성
4. 하위 우선순위 기준정보 목록을 필터링한다:
   a. 전체 옵션에서 rst_* 대상을 제거 (disabled)
   b. req_* 대상이 있으면 해당 옵션만 표시 (required)
5. 현재 선택이 제약 위반인지 검증한다
   -> ConstraintViolation[] 반환
6. 필터링된 결과를 UI에 반영한다
```

**rst_awkjob의 이중 레이어**: sizeinfo의 rst_awkjob은 규격 크기 범위 기반으로 "이 규격에서 이 후가공은 물리적으로 불가"를 판단하고, paperinfo의 rst_awkjob은 "이 지류에서 이 후가공은 재질 호환성 문제로 불가"를 판단한다. 두 레이어를 모두 평가해야 완전한 필터링이 가능하다.

### 5.5 NCR지 특수 규칙

NCR지(감압복사지)는 다층 구조(상지/중지/하지)로, 별도 상품으로 분류된다. 위젯 빌더 MVP에서는 우선순위를 낮추는 것을 권장한다.

## 6. 가격 계산 모델 (Pricing Model)

### 6.1 가격 조회 API

| 항목 | 값 |
|------|------|
| URL | `POST https://api.wowpress.co.kr/api/v1/ord/cjson_jobcost` |
| 입력 | 모든 기준정보 조합을 포함하는 주문 JSON |
| 출력 | 기준정보 조합별 단가 |

### 6.2 가격 결정 요소 (8가지)

가격은 아래 기준정보 조합에 의해 결정된다. 누락된 항목이 있으면 가격 산출이 불가하다.

1. **prodno** (상품번호) - 상품별 기본 가격 체계
2. **jobpresetno** (인쇄방식) - 최상위 가격 분기
3. **sizeno** (규격) - 크기별 가격 차이
4. **paperno** (지류) - 종이 종류/평량별 가격
5. **colorno + colornoadd** (도수) - 색상수별 가격
6. **ordqty** (수량) - 수량별 단가 차등 (비선형)
7. **awkjob** (후가공) - 후가공 추가 비용
8. **optno** (옵션) - MD 옵션 추가 비용

### 6.3 수량별 비선형 가격 구조

핵심 특징: 수량이 많을수록 단가가 낮아지는 비선형 구조이다.

- **합판 제품**: 연(500매) 단위, 0.5연~200연 범위
- **독판 제품**: 부/매 단위, min/max/interval로 범위 지정
- **동일 상품이라도** 지류/규격 조합에 따라 최소/최대 수량이 달라짐

### 6.4 가격 구성 요소 (Price Breakdown)

| 항목 | API 필드 | 설명 |
|------|---------|------|
| 인쇄비 | prscost | 기본 인쇄 비용 |
| 후가공비 | awkcost | 후가공 추가 비용 |
| 부자재비 | prodaddcost | 케이스, 거치대 등 |

| 할인 | dccost | 할인 금액 |
| 부가세 | taxcost | VAT 10% |
| 총 결제금액 | totalcost | 위 항목 합산 |

실제 가격 예시 (일반명함 500매, 스노우지 219g, 양면 칼라8도): 인쇄비 3,300원 + 부가세 330원 = 총 3,630원

## 7. 주문 플로우 (Order Flow)

### 7.1 API 기반 주문 프로세스

```
1. 인증 (JWT 토큰 발급)
   POST /api/login/issue {authUid, authPw}
   -> accessToken 발급 (Bearer)

2. 제품 목록 조회
   GET /api/v1/std/prodlist
   -> 전체 상품 목록 + 카테고리

3. 제품 상세 조회
   GET /api/v1/std/prod_info/{prodno}
   -> 전체 옵션 트리 JSON (coverinfo ~ deliverinfo)

4. 옵션 선택 (기준정보 우선순위에 따라 순차 선택)
   인쇄방식 -> 규격 -> 지류 -> 옵션 -> 도수 -> 후가공 -> 수량

5. 가격 조회
   POST /api/v1/ord/cjson_jobcost
   -> 기준정보 조합별 단가 반환

6. 주문
   POST /api/v1/ord/cjson_order
   -> 주문번호 발급

7. 파일 업로드
   POST file.wowpress.co.kr (Presigned URL 방식)
   -> AI/EPS/PDF 등 파일 업로드

8. 주문 상태 콜백
   주문상태 변경 시 CALL BACK URL로 통보
```

## 9. 위젯 빌더를 위한 시사점

### 9.1 필수 데이터 모델 (10개)

| 모델 | 설명 | JSONB 활용 |
|------|------|---------|
| ProductCategory | 계층형 카테고리 (최대 3단계) | keywords[] |
| PrintProduct | 상품 마스터 | coverInfo, rawData |
| ProductSize | 규격 + 비규격 범위 + 제약 | reqWidth, reqHeight, rstAwkjob |
| ProductPaper | 지류 + 평량 + 제약 | rstPrsjob, rstAwkjob |
| ProductColor | 도수 + 추가도수 + 제약 | reqPrsjob, rstAwkjob |
| ProductPrintMethod | 인쇄방식 | prsjobList, reqColor |
| ProductPostProcess | 후가공 그룹/작업 계층 | jobGroupList (깊은 중첩) |
| ProductOrderQty | 수량 (공통형/조합형) | qtyList[] |
| PricingTable | 기준정보 조합별 가격 | - |
| DeliveryInfo | 배송 정보 | freeShipping, methods, regions |

### 9.2 옵션 엔진 구현 핵심

**최소 필수 기능**:

1. 기준정보 우선순위 체인에 따른 순차 필터링
2. req_*/rst_* 제약 조건 실시간 평가
3. 공통형/조합형 수량 자동 결정
4. 비규격 크기 입력 범위 검증
5. 후가공 그룹 계층 탐색 및 제약 평가
6. req_joboption (라운딩 모서리 선택 등) 추가 입력 처리

**구현 전략**:

- 옵션 엔진은 **순수 TypeScript**로 구현 (Node.js + 브라우저 양립)
- 제약 조건은 **JSONB**로 저장, 애플리케이션 레벨에서 평가
- 후가공은 가장 복잡하므로 별도 모듈로 분리
- rst_awkjob은 **규격 레이어 + 지류 레이어** 이중 평가 필수

### 9.3 MVP 최소 옵션 셋 (단계별)

**1단계 MVP (핵심 4종)**:
- 명함: 규격(3) + 도수(2) + 지류(17) + 수량(조합형) = 단순 조합
- 스티커(사각): 규격(다수) + 수량(공통형) = 가장 단순
- 전단: 규격(14) + 도수(2) + 지류 + 수량(연) = 조합형
- 봉투: 규격 + 도수 + 지류 + 수량

**2단계 확장**:
- 후가공 옵션 (오시, 미싱, 타공, 박, 형압, 라운딩, 코팅)
- 책자류 (표지/내지 분리, 제본)
- 비규격 처리

**3단계 고급**:
- 사인제품 (직접입력 규격)
- NCR지 특수 처리
- 어패럴 (전혀 다른 옵션 체계)

### 9.4 WowPress 대비 간소화 포인트

| WowPress | 후니프린팅 위젯 | 이유 |
|---------|-------------|------|
| 326개 상품, 47개 카테고리 | 초기 20~50개 상품 | MVP 범위 축소 |
| 100~400 hidden field | React 상태 관리 | SPA 방식으로 전환 |
| 서버 사이드 가격 계산 | 클라이언트 옵션 엔진 + 서버 가격 확인 | 응답 속도 개선 |
| AJAX + jQuery | React/Next.js + REST API | 현대적 스택 |
| 고정폭 1,400px | 반응형 디자인 | 모바일 우선 |
| 후가공 2단 SELECT | 시각적 후가공 선택기 | UX 개선 |
| CSRF + 세션 | JWT + API 기반 | 위젯 SDK 호환 |

### 9.5 핵심 기술 결정사항

1. **JSONB vs 정규화**: 기준정보 최상위(sizeno, paperno, colorno)는 정규화, 제약조건(req_*/rst_*)과 후가공 상세는 JSONB
2. **클라이언트 옵션 엔진**: packages/pricing-engine을 순수 TypeScript로 구현하여 브라우저에서 직접 실행
3. **가격 검증**: 클라이언트에서 1차 계산, 서버에서 최종 확인 (2중 검증)
4. **데이터 크기**: 단일 제품 옵션 데이터가 수백KB 가능 (특수지명함 40070.json은 381KB). 위젯 SDK에는 필수 데이터만 전달 (50KB 미만 목표)

## 11. UI 패턴 벤치마킹

### 11.1 WowPress 사이트 기술 스택

- jQuery 3.6.0 + Bootstrap 5.1.3
- Spring 기반 서버 사이드 렌더링 (SSR) + AJAX 부분 갱신
- 고정폭 1,400px 레이아웃 (반응형 아님)
- 별도 모바일 리다이렉션

### 11.2 핵심 UI 패턴

1. **Cascading SELECT 드롭다운**: 모든 주요 옵션은 `<select>` 드롭다운, `onchange` 핸들러가 AJAX 호출 트리거
2. **실시간 AJAX 가격 계산**: 어떤 옵션이든 변경 시 자동 가격 재계산
3. **후가공 2단 SELECT**: 각 후가공 그룹은 namestep1(수량/재질) + namestep2(방향/크기) 구조
4. **우측 고정 가격 패널 (Sticky)**: 스크롤해도 항상 보이는 가격 요약
5. **Hidden Field 상태 관리**: 한 페이지에 100~400개 이상의 hidden input

### 11.3 상품별 UI 차이 비교

| 요소 | 명함 | 스티커 | 전단 | 책자 | 현수막 |
