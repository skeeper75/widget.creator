# 왜 위젯빌더인가?

인쇄 비즈니스의 특수성과 위젯빌더 접근법의 차별점을 이해합니다.

## Shopby를 선택한 이유

> "Shopby를 사용하게 된 것은 고도화 쇼핑플랫폼이 있다면 엔진 위에 다양한 형태의 비즈니스를 설계할 수 있을 거라는 기대였다. 그것이 인쇄라는 비즈니스였고, 와우프레스의 API Docs를 참고해서 위젯빌더로 어떤 쇼핑플랫폼이라도 바로 인쇄가 가능한 형태를 생각했다."

### 고도화 쇼핑플랫폼 엔진 위의 비즈니스 설계

| 장점 | 설명 |
|------|------|
| **확장성** | 플랫폼 엔진 위에 다양한 비즈니스 로직 추가 가능 |
| **안정성** | 검증된 이커머스 인프라 활용 |
| **통합성** | 회원, 장바구니, 주문, 결제 기능 기본 제공 |
| **멀티채널** | 하나의 상품 관리로 여러 채널 동시 판매 |

### 인쇄 비즈니스의 특수성

<div className="info-block">

인쇄 주문은 일반 이커머스와 다른 복잡성을 가집니다:

- **옵션 복잡성**: 용지 x 사이즈 x 수량 x 인쇄도수 x 후가공 조합
- **실시간 견적**: 옵션 변경 즉시 정확한 가격 계산 필요
- **생산 연동**: 주문 → 생산 지시 → 배송 파이프라인 자동화
- **다채널 통합**: Shopby, 자사몰, 제3자 사이트 연동

</div>

## 플랫폼 독립적 위젯 접근법

### 기존 방식 vs 위젯빌더

| 기준 | 와우프레스 방식 | huni.builder 위젯빌더 |
|------|---------------|---------------------|
| **배포** | 자체 사이트 전용 | 어떤 사이트에도 임베드 |
| **CSS 격리** | 없음 (호스트 페이지 의존) | Shadow DOM 완전 격리 |
| **가격 계산** | 서버 사이드 API 호출 | 클라이언트 내장 엔진 (실시간) |
| **다채널** | API만 제공 | 위젯 + API + 이벤트 프로토콜 |
| **번들 크기** | N/A (자체 사이트) | 48.71KB gzipped |
| **테스트** | N/A | 93.7% 커버리지 |

## 와우프레스 벤치마킹

<div className="tip-block">

**와우프레스(wowpress.co.kr)에서 참고한 핵심 사항**

- **API 구조**: 17개 Open API 엔드포인트 (회원/제품/가격/주문/파일/배송)
- **가격 체계**: 규격 x 도수 x 재질 기반 가격 조회 구조
- **옵션 설계**: 7개 옵션 영역 (규격, 도수, 재질, 후가공, 부자재, 배송, 수량)
- **제약 조건**: 선불택배불가, 묶음배송불가 등 제약 패턴

</div>

### 개선점 요약

huni.builder가 와우프레스 대비 개선한 사항:

- **실시간 클라이언트 가격 계산**: 서버 API 호출 불필요
- **Shadow DOM CSS 격리**: 어떤 사이트에도 충돌 없이 임베드
- **다채널 지원**: Shopby, 자사몰, 제3자 사이트 동시 지원
- **결정 테이블 기반 유연한 제약 조건**: 14개 패턴 템플릿
- **93.7% 테스트 커버리지**: 신뢰성 보장
- **TypeScript 타입 안전**: Typia/Nestia/Zod 전 계층

## 비전 로드맵

### 현재

- 위젯빌더 + 실시간 견적 + Shopby 연동
- 8개 계산기, 93.7% 커버리지
- MES/Edicus 연동 완료

### 다음 단계

- PDF/파일 주문 기능
- Edicus 편집기 통합 고도화
- MES 생산 자동화 확장

### 미래 비전

> "위젯빌더는 다양한 인쇄 도메인 지식이 결합되어 복잡하게 연결되기 때문에 이를 UI/UX로 표현하는 것이 어렵다. 네트워크 토폴로지나 N8N, BPS와 같이 Node형태로 의미를 함께 부여하고, AI와 함께 상품을 설계해 나가는 형태까지 발전하려는 것이 목적이다."

- **AI 기반 상품 설계**: 고객 데이터 학습, 상품 기획 자동화
- **노드 기반 UI/UX**: N8N/BPS 스타일 비주얼 에디터
- **도메인 지식 시각화**: 복잡한 옵션 관계를 그래프로 표현
